# 联调清单（按这个顺序，最不容易卡死）

> 目的：让“接口都写完但合不上”的风险最低。
> 原则：先跑最短链路，再逐步替换 mock 为真实实现。

---

## A. 环境就绪
- `docker-compose.yml` 能启动：Postgres、Redis、MinIO
- Go API 能启动：`/health` 返回 ok
- 教师端/学生端能启动：能访问页面

---

## B. SSE 最短链路（Go → 学生端）
1. Go 新增（或临时）SSE 接口：固定推 10 个 `token` + 1 个 `final`
2. 学生端用 EventSource/Fetch SSE 能看到打字机效果

验收：
- 页面能持续收到 token
- final 到达后 UI 状态变为“可续接”

---

## C. Python 问答流（Python → Go，不接前端）
1. Python 端提供一个本地可调用的 QA stream（HTTP 或 CLI）
2. Go 端能读到流并打印/转发

验收：
- Go 端能逐段读到 token
- 结束能拿到 `need_reteach/resume_*` 字段

---

## D. 全链路问答（学生端 → Go SSE → Python → Go SSE → 学生端）
验收：
- 输入问题后 2 秒内开始出 token（体感）
- `final` 结构化字段在 UI 可见（可隐藏，但必须可打印调试）

---

## E. 续接（Redis Session）
1. 学生端上报 `current_page/current_node_id`
2. 问答开始：Go 将 session 标记 interrupted
3. 问答结束：Go 写回 resume_*，学生端恢复播放

验收：
- 能复现：在 node A 打断问答 → 回答完 → 继续 node A+1

---

## F. 解析链路（上传 → 队列 → 解析 → 入库 → 教师端查看）
验收：
- 上传后状态为 parsing
- 解析完成变为 ready
- 教师端可按页查看脚本并保存编辑
