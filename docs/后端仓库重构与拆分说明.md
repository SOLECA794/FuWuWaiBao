# 后端架构拆分整理文档

## 1. 修改背景与逻辑
为了提高代码的可维护性和模块化程度，我们将原有的单一后端服务（Monolith）拆分为 **学生端 (Student Server)** 和 **教师端 (Teacher Server)** 两个独立的微服务入口。

### 修改逻辑：
1.  **入口分离**：在 `backend/cmd/` 下创建了 `student/` 和 `teacher/` 目录，分别存放各自的 `main.go`。这允许两端拥有独立的生命周期、日志输出和端口配置。
2.  **路由解耦**：在 `backend/internal/router/` 下创建了 `student.go` 和 `teacher.go`。通过函数式注册（`RegisterStudentRoutes` / `RegisterTeacherRoutes`）将复杂的路由定义从主程序中剥离。
3.  **共享内核**：保持 `internal/handler`、`internal/service` 和 `internal/model` 层的代码共享，确保业务逻辑的一致性，同时避免代码冗余。
4.  **环境独立**：学生端默认运行在 `8081` 端口，教师端默认运行在 `8082` 端口，便于前端和 Nginx 进行流量分发。

---

## 2. 影响范围及后续调整建议

### 2.1 端口变更
*   **学生端**：`http://localhost:8081`
*   **教师端**：`http://localhost:8082`

### 2.2 前端相关修改
必须更新前端项目中的 API 基础 URL 配置：
- **学生端前端 (`frontend/student`)**：
  - 修改 `src/config/api.js` 或 `.env` 文件，将 `VITE_API_BASE_URL` (或类似变量) 指向 `http://localhost:8081`。
- **教师端前端 (`frontend/teacher`)**：
  - 修改相应的配置文件，指向 `http://localhost:8082`。

### 2.3 运维与脚本
- **Docker 部署**：需要更新 `docker-compose.yml`，为两个新服务分别配置容器和端口映射。
- **启动脚本**：`scripts/start-all.ps1` 需要更新，以分别启动这两个 Go 程序而不是原来的单一服务。

## 3. 代码贡献量估计 (基于 Handler 行数统计)

根据对 `backend/internal/handler/` 目录下核心业务代码的行数统计：

| 模块 | 文件 | 行数 (LOC) | 占比 |
| :--- | :--- | :--- | :--- |
| **学生端 (Student)** | `student.go` + `compat_student.go` | 473 + 401 = **874** | **61.3%** |
| **教师端 (Teacher)** | `teacher.go` + `compat_teacher.go` | 357 + 206 = **563** | **38.7%** |
| **总计** | - | **1437** | 100% |

### 贡献分析：
1.  **学生端 (约 61%)**：
    *   **复杂度高**：包含了大模型问答、追问逻辑、知识点弱点诊断以及复杂的会话状态维护。
    *   **交互密集**：支持 SSE 流式输出和多维度的学习数据统计，代码量最大，是项目的核心用户价值所在。
2.  **教师端 (约 39%)**：
    *   **基础性强**：侧重于课件上传、解析分发、脚本编辑和班级数据概览。
    *   **逻辑清晰**：多为 CRUD 和 异步处理集成，虽然行数较少，但承担了系统的数据源头管理工作。

> **结论**：从代码规模和逻辑复杂度看，学生端的开发工作量约为教师端的 **1.5 倍**。
