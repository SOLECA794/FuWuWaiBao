# 播放时间轴与语音同步设计

## 1. 当前已落地能力

- 学生端已经按页获取节点化脚本。
- 后端 TeachingNode 已补充预计时长字段，可返回每个节点的 duration_sec、start_sec、end_sec。
- 学生端已支持基于时间轴的最小自动播放：
  - 播放时按节点预计时长自动推进。
  - 问答结束后可恢复到指定节点继续。
  - 到达页末后自动切换到下一页。

这一版属于可演示的伪同步播放，目标是先把讲授节奏、节点高亮和续接体验建立起来。

## 2. 为什么这一版先不直接做真实 TTS 同步

当前仓库里：

- OpenAPI 的 generateAudio 仍是占位返回。
- 没有真正的音频文件生成、存储、时长回填和分段 metadata。
- 学生端也没有真实音频播放器状态与节点进度的绑定。

如果直接上真实 TTS，会同时牵涉：

- 语音合成供应商接入
- 音频文件存储与访问
- 音频总时长与节点时长切分
- 前端播放器事件和时间轴同步

这会把当前任务从补演示主链路拉成补完整音频系统。因此这一轮先做最小闭环更合适。

## 3. 推荐的真实 TTS 同步架构

### 3.1 数据层

建议给 TeachingNode 补充：

- estimated_duration
- audio_url
- audio_duration_sec
- audio_start_sec
- audio_end_sec
- tts_status

建议新增 AudioAsset：

- id
- course_id
- page_index
- node_id
- voice_type
- audio_url
- duration_sec
- format
- status

## 3.2 后端接口层

建议新增内部接口：

- POST /api/v1/teacher/coursewares/{courseId}/audio/generate
- GET /api/v1/teacher/coursewares/{courseId}/audio/status
- GET /api/v1/student/coursewares/{courseId}/audio/{pageNum}

返回结构应包含：

- page_audio_url
- total_duration_sec
- nodes: [{ node_id, start_sec, end_sec, audio_url }]

## 3.3 前端播放层

学生端需要真正引入 audio element 或播放器实例，并把以下事件接进来：

- loadedmetadata
- timeupdate
- play
- pause
- ended

同步规则：

- timeupdate 时根据 currentTime 命中当前节点。
- 学生手动点节点时，播放器 seek 到 node.start_sec。
- 问答结束返回 resume_node_id 时，播放器 seek 到对应 start_sec 并自动恢复。

## 4. 推荐分阶段实现顺序

### 阶段 A

- 保留当前预计时长驱动的时间轴播放
- 教师端补预计讲授时长展示
- 学生端补当前节点剩余时间显示

### 阶段 B

- 接入真实 TTS 供应商
- 生成整页或整节点音频
- 回填节点起止时间

### 阶段 C

- 语速调节
- 追问后无缝 seek 恢复
- 支持重讲脚本的单独音频片段

## 5. 结论

当前版本已经把节点时间轴、自动讲授节奏和续接到节点做成了一版可展示能力。

下一轮如果继续做，就应该沿着真实音频 metadata 驱动同步这条线推进，而不是再回到纯脚本播放。
