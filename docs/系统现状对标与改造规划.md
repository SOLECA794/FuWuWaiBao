# 系统现状对标与改造规划

## 1. 结论先行

结合现有代码、现有文档以及对话记录里的参考判断，可以先下一个更准确的结论：

- 这套系统不是“方案阶段”，而是已经具备一条可运行的 MVP 主链路。
- 参考对话里提到的部分“缺失项”现在已经补上了，尤其是教师脚本编辑、节点化脚本、学生播放脚本接口、断点续学、基础学情统计这几块。
- 但系统目前仍然明显偏“单课件演示系统”，还没有真正升级成“多角色、多课程、多班级、多会话”的平台型产品。
- 如果目标是交付 PPT 和演示视频，当前系统已经足够支撑第一版演示；如果目标是对外讲“完整平台”，则需要补上账号体系、课程体系、多轮对话状态、播放时间轴和更可信的学情聚合。

一句话概括：

当前系统已经覆盖了“课件解析 -> 内容重构 -> 基础讲稿编辑 -> 学生播放 -> 实时答疑 -> 页级续接”的主流程，但还没有完整覆盖“多轮问答会话 -> 节奏自适应 -> 平台级用户课程体系 -> 可审计的学情分析”。

## 2. 现有系统里已经实现的内容

以下能力已经不是停留在文档，而是代码层面已经落地。

### 2.1 智课生成主链路已实现

- 教师上传课件后，后端会保存课件、解析页面内容、落库课件页，并调用 AI 引擎做内容重构。
- 重构结果不是单纯一页一段文本，而是已经拆成 TeachingNode 教学节点。
- 每个节点已经包含标题、摘要、核心点、示例、常见困惑、重讲脚本、过渡语等字段，为后续讲授和续接留了结构空间。

对应实现：

- backend/internal/service/course.go
- backend/internal/model/models.go
- backend/internal/service/ai_engine_client.go

### 2.2 教师脚本编辑功能已实现

参考对话里把“教师脚本编辑”列为致命缺口，但当前仓库里这项能力已经落地：

- 教师端已有讲稿编辑页面。
- 后端已有获取讲稿、保存讲稿、AI 重新生成讲稿接口。
- 脚本编辑既支持直接编辑页级 ScriptText，也支持基于 TeachingNode 生成再保存。

对应实现：

- backend/internal/handler/teacher.go
- frontend/teacher/src/App.vue
- docs/API_DESIGN_V2.md

### 2.3 学生端播放脚本与节点展示已实现

- 学生端不是只拿一整页文本，而是会拿 nodes 列表。
- 后端已经支持按页返回 playback nodes 和 page_summary。
- 前端已经支持展示节点列表、切换节点、页级播放状态和课件预览。

这意味着“节点化讲授”已经比参考对话里判断的更成熟，只是还缺时间轴与音频同步。

对应实现：

- backend/internal/handler/student.go
- backend/internal/handler/compat_student.go
- backend/internal/handler/teaching_node_support.go
- frontend/student/src/App.vue

### 2.4 实时问答与续接基础链路已实现

- 学生端已有 SSE 问答流接口。
- 返回结果已包含 need_reteach、source_page、resume_page、resume_node_id。
- 前端收到 final 事件后，已经会更新当前节点并维护最近问答历史。

这说明“打断提问 -> AI 回答 -> 返回续接位置”的骨架是存在的。

对应实现：

- backend/internal/handler/student.go
- backend/internal/handler/compat_student.go
- frontend/student/src/App.vue

### 2.5 基础学情、断点、笔记、薄弱点能力已实现

- 教师端已有 stats、questions、card-data 三类分析入口。
- 学生端已有 breakpoint、notes、stats、weak-points、knowledge-graph 等接口。
- 薄弱点、知识点树、习题生成这些能力甚至已经超出参考对话里最初的 MVP 范围。

对应实现：

- docs/当前运行接口清单.md
- docs/API_DESIGN_V2.md
- backend/internal/model/models.go

### 2.6 泛雅适配层接口框架已实现

- 已有开放适配层路由。
- 已有 lesson、qa、progress、platform 这几类开放接口入口。
- 已支持 enc + time 的签名校验框架。

这部分说明系统已经有“轻量集成泛雅”的技术入口，不是纯内部系统。

对应实现：

- backend/api/main.go
- backend/internal/handler/compat_openapi.go
- docs/API_DESIGN_V2.md

## 3. 相比参考对话，当前系统其实做得更好的地方

参考对话里的分析大方向没有错，但它低估了当前仓库已经做完的几件事。

### 3.1 已经不是“只有页级脚本”，而是“节点级教学结构”

参考对话把系统理解成“有脚本但没有讲授结构”，这在当前代码里不完全成立。

当前系统里已经有：

- TeachingNode 模型
- node_id
- source_pages
- core_points
- reteach_script
- transition_text
- interactive_questions

这说明系统底层数据已经朝“可讲授、可续接、可重讲”的结构化内容演进，而不只是生成一大段讲稿。

### 3.2 已经支持教师编辑，而不是纯 AI 黑盒输出

参考对话认为“脚本可编辑”没有落地，但当前教师端已经具备编辑和保存的闭环。这个点对演示很重要，因为它让产品从“AI 自动生成器”变成“教师可控的教学工具”。

### 3.3 已经有学生学习侧的衍生能力

当前系统除了问答，还补了：

- 断点恢复
- 学习统计
- 薄弱点诊断
- 笔记
- 知识点拆解
- 习题生成与校验

这些能力虽然实现深度不一定都很强，但从演示角度已经能把学生端包装成“AI 学伴”而不是单一问答窗口。

### 3.4 已经有开放适配层，不只是内部前后端直连

对于“泛雅轻量集成”这个目标，当前方案的接口组织明显优于单一内部接口设计，因为已经把内部 API 和外部 OpenAPI 适配层分开了。这个思路是对的，后续应该保留。

## 4. 当前系统真正还缺什么

这里要区分“有接口/有字段”和“体验闭环真的成立”。当前最核心的缺口有六类。

### 4.1 缺真正的多轮对话状态

现在的问答虽然有 qaHistory，但主要停留在前端最近几条记录展示。

当前问题：

- 后端 AskWithContext 只接收 question、current_page、context，没有 session 级历史摘要。
- AI 侧 QAResponder 也没有真正接收 conversation summary 或多轮 messages。
- need_reteach 基本还是靠关键词触发，不是基于历史上下文判断。

结果：

- “再举个例子”“换个说法”“刚才第二点是什么意思” 这类追问，还没有真正依赖前文语义链。

### 4.2 缺播放时间轴和音频同步能力

当前学生端虽然有 nodes，但还没有完整的 PlaybackService。

当前问题：

- node 里没有 duration_hint、audio_start、audio_end 之类元数据。
- 前端 isPlay 更多是本地播放状态切换，不是真正驱动音频时间轴。
- 还没有“按 node 恢复到某个时间点继续播放”的能力。

结果：

- 现在的“讲授”更像脚本浏览，不是可严格演示的语音同步讲授。

### 4.3 缺更可信的节奏调整

当前系统里虽然有 need_reteach、understandingLevel 这些字段，但核心判断仍然比较浅。

当前问题：

- understandingLevel 本质上仍由 need_reteach 映射得出。
- 没有独立的理解程度评分逻辑。
- 没有根据学生历史提问、停留时长、重复访问、练习结果动态调整讲解深度。

结果：

- “智能节奏调整”目前更适合在方案里讲，暂时还不适合作为强验收点。

### 4.4 缺平台级用户和课程体系

这部分是当前系统和“独立平台”之间最大的差距。

当前问题：

- 没有 User、Teacher、Student、Role、CourseClass、Enrollment 这类真正的平台实体。
- 目前 Course 更接近“课件”而不是“课程”。
- 一个老师多门课、一个学生多门课、多班级归属，这些都还没有模型支撑。
- syncUser、syncCourse 有开放接口入口，但内部没有形成完整领域模型。

结果：

- 现在适合讲“AI 互动智课系统”，不适合直接讲“完整教学生态平台”。

### 4.5 学情分析仍偏轻量统计，缺聚合分析层

当前教师 stats、questions、card-data 已经有接口，但更接近简单统计和日志聚合，还不是成熟学情引擎。

当前问题：

- 没有统一的 LearningEvent 事件流模型。
- 没有知识点级热度聚合。
- 没有按班级、课程、学生维度的分析面板。
- 没有风险学生预测或掌握度计算。

结果：

- 现在能支持演示“有分析”，但还不足以支撑“学情诊断平台”的强叙事。

### 4.6 教师编辑缺版本化和发布流

虽然脚本编辑已经存在，但它还是偏单版本直接覆盖。

当前问题：

- TeacherEdit 有记录，但没有真正形成草稿版、已发布版、历史版本对比。
- 没有页级/节点级的审核与发布状态。
- 编辑结果和学生端播放脚本之间缺少明确的发布界线。

结果：

- 当前适合单人演示，不适合多人协作或正式教学使用。

## 5. 对参考方案的吸收建议

参考对话里的思路可以吸收，但要做成“贴合当前代码基线”的版本，不要推翻重来。

### 5.1 保留当前 TeachingNode 架构，不要退回页级脚本设计

后续所有改造建议都应该围绕 TeachingNode 扩展，而不是重新设计成纯 page-script。

建议新增字段：

- estimated_duration
- audio_url
- audio_duration_ms
- resume_offset_ms
- difficulty_level
- mastery_hint

### 5.2 把“多轮问答”做成 Session，而不是前端临时历史

建议新增：

- DialogueSession
- DialogueTurn
- ContextSummary

后端做法：

- 每次提问带 sessionId
- 后端按 sessionId 拉最近 N 轮或摘要
- 再把摘要拼接到 AI prompt

### 5.3 把“节奏调整”拆成两个可落地层次

第一层先做规则版：

- 连续追问两次同页同节点
- 同一页停留过久
- 同一知识点练习错误率高

第二层再做模型版：

- LLM 或分类模型输出 understandingLevel
- 结合行为日志做自适应推荐

这样更适合当前项目节奏。

### 5.4 学情分析先做事件表，再做可视化

不要先堆复杂图表，应该先统一事件数据。

建议新增 LearningEvent：

- course_id
- user_id
- session_id
- page_index
- node_id
- event_type
- duration_ms
- payload_json

有了这层，后续才好做：

- 热点页
- 热点知识点
- 续接成功率
- 平均停留时长
- 学生风险预警

### 5.5 平台化不要一步到位，先补最小课程域模型

建议第一阶段只新增：

- User
- Course
- Courseware
- Enrollment

并将现有 Course 重命名或语义重定位为 Courseware，再由新的 Course 关联多个 Courseware。

## 6. 推荐改造路线

### 阶段 A：演示闭环增强

目标：让 PPT 和演示视频更稳，更像一个完整产品。

优先级最高的改造：

1. 多轮问答 Session 化
2. 节点播放元数据补齐
3. 续接提示与恢复体验优化
4. 教师端脚本编辑增加节点视图
5. 学情页补充热点页、热点知识点、续接率三项核心指标

完成后能支撑的演示脚本：

- 教师上传课件
- AI 生成节点化讲稿
- 教师微调脚本
- 学生播放讲授并提问
- AI 多轮追问解释
- 自动回到原节点继续
- 教师查看热点和卡点分布

### 阶段 B：平台最小闭环

目标：让系统从“单课件演示工具”升级到“可运营平台”。

建议新增：

- 用户登录与角色
- 课程与选课关系
- 班级或教学班
- 课件挂载到课程
- 学生个人课程列表

这一步完成后，系统才能自然回答：

- 谁在学
- 学的是哪门课
- 属于哪个老师
- 统计面板该按谁聚合

### 阶段 C：智能诊断增强

目标：把“AI 互动智课”升级成“AI 学习诊断系统”。

建议新增：

- Knowledge mastery scoring
- 风险学生预警
- 个性化讲解深度
- 推荐练习与复习路径

这部分更适合做第二轮亮点，而不是第一版交付刚需。

## 7. 我建议你们下一步真正先做什么

如果以“近期要交 PPT 和演示视频”为目标，我建议不要马上全面平台化，而是先把下面 4 个点做扎实。

### 第一优先级

1. 多轮问答 Session
2. 节点播放时间元数据
3. 教师端节点级脚本编辑
4. 教师学情页补三个真实指标

这 4 项能同时提升：

- 演示流畅度
- 需求贴合度
- 官方需求覆盖率
- 后续平台化扩展空间

### 第二优先级

1. 用户与课程模型
2. 课程与课件解耦
3. 泛雅用户/课程同步落到真实实体

### 第三优先级

1. 自适应节奏算法
2. 风险预警
3. 个性化学习路径

## 9. 最终判断

如果问题是“我们目前的系统设计能不能符合参考对话里的目标”，答案是：

- 能覆盖主链路，但不是 100%。
- 已经比参考对话里判断的更完整。
- 当前最适合的动作不是推翻重做，而是基于现有架构做一次聚焦式增强。

如果以交付优先级来排序：

- 现在系统足够做第一版展示。
- 经过一轮针对性补强后，可以比较稳地支撑正式答辩和产品演示。
- 再往后，才值得进入完整平台化改造。
