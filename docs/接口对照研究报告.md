## 2. 结论摘要

### 2.1 总体判断
当前项目已经形成了一套**可运行的内部 API**，但存在以下明显特征：

1. **教师端接口完成度较高**，且大部分已与 `docs/后端API.md` 对齐。
2. **学生端接口已实现了一批实际可调用接口**，但与文档中的设计版接口路径、方法、字段存在较大偏差。
3. **`API_DESIGN_V2.md` 中的 `/api/v1/...` 体系目前大多尚未落地为真实路由**。
4. **泛雅开放接口（`/api/v1/lesson/*`、`/api/v1/qa/*`、`/api/v1/progress/*`）在 Go 后端中尚未实现**。
5. **项目中没有真正意义上的“前端本地 mock 接口服务”**，但存在若干**半 mock / 推导型接口**，即：
   - 后端在数据不足时返回默认值或估算值；
   - 前端在接口失败时使用兜底展示逻辑。

### 2.2 三类结果总览

#### A. 项目已实现且文档存在（以精确路径对照）
主要集中在教师端：
- `/health`
- `/api/teacher/courseware-list`
- `/api/teacher/upload-courseware`
- `/api/teacher/courseware/:courseId`
- `/api/teacher/publish-courseware`
- `/api/teacher/script/:courseId/:page`
- `/api/teacher/script/save`
- `/api/teacher/ai-generate-script`
- `/api/teacher/student-stats/:courseId`
- `/api/teacher/question-records/:courseId`
- `/api/courseware/:courseId/page/:pageNum`

> 注：`/health` 在代码中存在，但主文档中未系统登记；这里只作为项目实际存在的通用接口记录。

#### B. 文档存在但未实现（或未按文档路径/协议实现）
主要集中在：

## 6. Mock / 半 Mock 接口识别结果

### 6.1 纯 mock 接口

本次扫描结果：**未发现真正的前端本地 mock HTTP 接口**。  
即：项目中没有发现独立的 `mock server`、`mock.js`、`fake API`、`json-server` 之类实现。

### 6.2 半 mock / 推导型接口

以下接口虽然是真实后端接口，但在数据不足时会返回推导值、默认值或占位值，因此应归为“半 mock / 演示型实现”：

| 接口 | 类型 | 原因 |
|---|---|---|
| `/api/teacher/card-data/:courseId` | 半 mock | 无行为日志时，按页码推导停留时长与卡点指数 |
| `/api/teacher/student-stats/:courseId` | 半 mock | 关键词数量不足时补默认关键词（如“概念”“原理”等） |
| `/api/weakPoint/getList` | 半 mock | 无提问日志时，返回“暂无薄弱点”默认项 |

### 6.3 前端兜底展示逻辑（非接口，但会影响“是否像 mock”）

| 文件 | 位置 | 行为 |
|---|---|---|
| `frontend/student/src/App.vue` | `loadWeakPoints()` | 接口失败时仅告警，不强制失败，存在默认展示倾向 |
| `frontend/teacher/src/App.vue` | `loadStudentStats()` / `loadCardData()` | 接口失败时用空数据或失败文案兜底 |

## 10. 关键问题与风险点

### 10.1 路由体系分裂
当前至少并存 3 套接口风格：

1. **旧版内部接口**：`/api/teacher/*`、`/api/student/*`、`/api/weakPoint/*`
2. **设计版 REST 接口**：`/api/v1/...`
3. **AI 引擎内部接口**：`/generate-script`、`/ask-with-context`、`/parse-knowledge`

这会导致：
- 文档难以维护；
- 前后端联调时容易误用接口；
- 后续泛雅适配层难以平滑接入。

### 10.2 教师端与学生端对接口的复用不合理
学生端当前通过 `/api/teacher/courseware-list` 获取课件列表，说明：
- 学生端缺少独立的“已发布课件列表”接口；
- 角色边界尚未收敛；
- 后期权限控制会有问题。

### 10.3 SSE 设计已写入文档，但代码未落地
`docs/后端API.md` 与 `docs/联调清单.md` 都强调了：
- 问答流式返回；
- 打断与续接；
- `sessionId` / `nodeId` / `resumeNodeId` 等。

但当前代码仍是普通 JSON 问答接口，**没有 SSE 实现**。

### 10.4 学生端“设计接口”和“现有实现”差距最大
当前学生端虽然功能上已经能跑通：
- 提问
- 获取断点
- 获取学情
- 薄弱点
- 讲解
- 出题
- 判题

但接口设计层面与文档差异较大，后续如果要对外扩展或接泛雅平台，重构成本会集中爆发在学生端。

## 11. 建议的整改优先级

### P0：先补接口文档
建议优先把当前真实运行接口补入正式文档：
- `/api/teacher/card-data/:courseId`
- `/api/student/courseware/page`
- `/api/student/ai/question`
- `/api/student/ai/traceQuestion`
- `/api/student/studyData`
- `/api/student/breakpoint`
- `/api/student/saveNote`
- `/api/weakPoint/*`
- `/api/ai/parseKnowledge`
- AI 引擎内部接口（单独加“内部依赖接口”章节）

### P1：明确“当前运行接口”与“目标统一接口”
建议在文档中新增两层定义：
- **Current API（现网/现代码）**
- **Target API（目标统一版 `/api/v1`）**

这样可避免大家把设计稿误当成现有接口。

### P2：优先收敛学生端接口
优先将以下能力迁移到统一风格：
- `/api/student/ai/question` → `/api/v1/ai/coursewares/{courseId}/ask`
- `/api/weakPoint/getList` → `/api/v1/student/coursewares/{courseId}/weak-points`
- `/api/student/breakpoint` → `/api/v1/student/coursewares/{courseId}/breakpoint`

### P3：补开放适配层
若后续确实要对接泛雅，应在 Go 后端单独增加：
- `enc` / `time` 校验中间件
- `/api/v1/lesson/*`
- `/api/v1/qa/*`
- `/api/v1/progress/*`
- `/api/v1/platform/*`

避免直接把内部接口暴露给平台。

---

## 12. 最终结论

如果以“当前代码真实可调用接口”为准：
- **教师端接口基本成型，文档匹配度较高**；
- **学生端接口功能已初步闭环，但命名与路径仍偏旧版实现**；
- **统一 REST 版接口和泛雅开放接口仍主要停留在设计层**；
- **项目里没有真正的 mock server，但有多个半 mock / 估算型接口**。

因此，当前项目最准确的状态应表述为：

> **“已有一套可运行的内部老版接口；统一化 `/api/v1` 体系与泛雅开放适配层尚未完成落地。”**
