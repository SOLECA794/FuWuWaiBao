# 开发规划与任务拆解（基于当前代码现状）

> 更新时间：2026-03-06
> 适用对象：项目负责人、Go 后端、Python AI、前端教师端、前端学生端。
> 目标：基于“当前仓库真实代码”，明确项目已完成什么、未完成什么、接下来怎么推进，以及每个人应该接什么任务。
> 约定：接口以 [API_DESIGN_V2.md](API_DESIGN_V2.md)、[当前运行接口清单.md](当前运行接口清单.md) 和当前代码实现为准；规划以“代码真实状态优先”，不再以旧文档假设作为开发依据。

---

## 1. 先说结论

当前项目不是从零开始，而是已经进入“多人并行改动后，需要收敛结构和稳定交付”的阶段。

现在最重要的不是继续盲目加功能，而是先完成这三件事：

1. 统一当前真实可用的接口和目录结构，避免前后端继续各写各的。
2. 先补齐 MVP 闭环里真正缺失的能力，而不是继续堆开放接口骨架。
3. 把任务拆成 GitHub 上可执行、可验收、可指派的 issue，而不是继续保留宏观描述。

## 2. 当前代码现状（以代码为准）

### 2.1 当前真实目录

- Go 后端入口：`backend/api/main.go`
- Go 主要处理器：`backend/internal/handler/*.go`
- Go 服务层：`backend/internal/service/*.go`
- Python AI 引擎：`ai_engine/main.py`
- 教师端前端：`frontend/teacher`
- 学生端前端：`frontend/student`

说明：旧文档里提到的 `teacher-frontend`、`student-frontend`、`cmd/api` 已经不再对应当前主开发路径，不能继续作为任务分派依据。

### 2.2 当前已经落地的能力

- Go 后端已经注册了三套接口：旧兼容接口、`/api/v1` 统一接口、开放适配层接口。
- 课件上传、课件列表、讲稿读取/保存、AI 生成讲稿、学生问答、薄弱点、笔记、断点等接口均已有可调用实现。
- Python AI 引擎已提供：讲稿生成、上下文问答、知识点解析接口。
- 教师端和学生端前端都已经写了对接接口的服务层。
- Redis、Postgres、MinIO 已经纳入后端初始化流程。

### 2.3 当前尚未真正完成的能力

- 课件上传后的“异步解析队列”还没落地，当前上传只完成入库和存储，没有真正形成异步解析流水线。
- 学生 SSE 问答目前只是 Go 拿到完整 AI 回答后再拆成 token 推送，本质上不是真正的上游流式输出。
- 会话状态虽然有内存 + Redis 写入，但还没有形成稳定的学习进度状态机。
- 知识图谱和薄弱点目前依赖数据库已有数据，没有和“上传后自动解析写库”完全串起来。
- 开放适配层多数是可调用骨架，不是生产级闭环。

### 2.4 本轮已处理的实际问题

- 已修复 `backend/api/main.go` 中一批 `/api/v1` 路由缺少前导 `/` 的注册问题。否则教师端和学生端按规范路径请求时，后端可能无法命中。

## 3. 当前最真实的风险点

以下不是“以后再说”的问题，而是会直接影响多人协作效率的阻塞项。

| 风险 | 当前情况 | 影响 |
|---|---|---|
| 接口结构并存 | `/api` 旧接口、`/api/v1` 新接口、开放适配接口同时存在 | 前端和后端很容易对错版本 |
| 前端目录迁移 | 主前端已迁移到 `frontend/teacher` 和 `frontend/student` | 队友如果按旧目录开发会直接走错 |
| 学生端服务层混用旧接口和新接口 | 学生端源码同时调用 `/api/student/...` 和 `/api/v1/...` | 联调成本高，排查困难 |
| AI 流式是伪流式 | 目前是后端拆完整答案后推 SSE | 首 token 延迟、体验和架构都不符合目标 |
| 上传解析未串通 | 课件上传未触发真实解析任务和状态流转 | 教师上传后的业务闭环不完整 |
| 安全和鉴权仍是骨架 | CORS 放开、开放接口签名缺少时效校验、业务接口未真正鉴权 | 测试阶段可用，后续上线风险高 |

## 4. 推进策略：先稳住，再闭环，再增强

建议不要继续按“教师端、学生端、AI 各自埋头开发”推进，而是按以下 3 个阶段推进。

### 阶段 A：收敛与稳定（P0）

目标：把“项目能不能稳定联调”这个问题先解决。

必须完成：

- 路由、目录、文档、启动命令全部对齐当前代码。
- 前端统一使用一套明确的接口体系。
- 确认哪些接口是生产链路，哪些只是兼容或骨架。

### 阶段 B：跑通真实 MVP 闭环（P1）

目标：教师上传课件后，能形成最基本的可学习闭环。

闭环定义：

1. 教师上传课件。
2. 系统解析并生成页级内容。
3. 教师可编辑讲稿。
4. 学生可加载课件和脚本。
5. 学生可问答。
6. 问答后可恢复播放位置。

### 阶段 C：体验与平台化增强（P2）

目标：从“能跑”变成“能演示、能比赛、能交付”。

必须完成：

- 真实流式问答。
- 弱点诊断质量提升。
- 统计数据从占位变为真实行为数据。
- 开放适配层从骨架变成可验收接口。

## 5. 角色分工（按当前项目实际来分）

### 5.1 Go 后端

负责范围：

- 所有 HTTP 路由注册与兼容层收敛
- 数据库模型与落库逻辑
- 课件上传、状态、预览、讲稿持久化
- SSE 代理与会话状态
- 开放适配层

不建议继续承担：

- AI Prompt 细节
- 解析器内部规则
- 前端页面逻辑

### 5.2 Python AI

负责范围：

- 文档解析 schema
- 讲稿生成
- 上下文问答
- 知识点树生成
- 后续真实流式输出

必须对外稳定的契约：

- 讲稿返回结构
- 问答返回结构
- 知识点解析结构

### 5.3 教师端前端

负责范围：

- 课件管理
- 讲稿编辑
- 发布、学情、记录页面
- 明确展示后端状态而不是自己猜业务状态

### 5.4 学生端前端

负责范围：

- 课件列表与进入学习
- 脚本播放与节点高亮
- 问答输入和 SSE 展示
- 断点保存与续接
- 弱点诊断与练习流程

## 7. 项目推进规则（必须执行）

1. 不允许只在群里口头说“我做完了”，必须有 issue、提交记录和验收说明。
2. 任何接口改动，必须同步更新 [当前运行接口清单.md](当前运行接口清单.md) 或 [API_DESIGN_V2.md](API_DESIGN_V2.md)。
3. 前端开发前先确认用的是哪一套接口，不能默认猜旧接口还能长期存在。
4. AI 返回结构一旦改字段名，必须先发 PR 说明，否则前后端会一起炸。
5. 每个任务必须带 DoD（验收标准），否则不准进 Done。

## 8. 谁拿到这份文档后该做什么

### 对项目负责人

- 先建 GitHub Project。
- 按照新的 GitHub 拆解表建 issue。
- 把每个 issue 指派给具体同学。
- 每天只盯阻塞项，不盯“看起来很忙”。

### 对开发同学

- 不要从旧文档开始看，先看当前代码目录。
- 接任务前先确认依赖项有没有完成。
- 做完后用接口、页面、日志或录屏证明已完成。
