# 项目详细方案：需求分析、架构设计与算法流程

> 本文用于：研发实施指导 + 评审答辩材料。
> 文档定位：把“宏大概念”落成可执行模块、可对接接口、可验收指标。

---

## 1. 需求分析

### 1.1 业务目标
- 将 PPT/PDF 静态课件自动转化为结构化可讲授内容（脚本+导图+重点）。
- 支持学习过程中的实时问答（文字/语音），答案与课件高度相关。
- 支持问答后的进度续接，并根据理解程度自动调节讲授节奏。

### 1.2 角色与核心诉求
- 教师：上传课件→自动生成讲稿→可编辑→发布；希望减少录课成本。
- 学生：听讲时可随时打断提问；答疑后能继续讲、不丢进度。
- 平台（泛雅）：标准化开放 API、签名校验、异步任务对接、可扩展字段。

### 1.3 功能性需求（按闭环）
1) **课件解析**：PPT/PDF → 页级结构化内容（含公式/图表要点、知识点层级）。
2) **智课生成**：生成“开场白-讲解-过渡语”的结构化脚本，支持教师编辑。
3) **实时问答**：携带当前学习上下文（页/章节/节点）进行 RAG 问答，多轮对话。
4) **进度续接**：问答后回到原节点；若未理解则补充/重讲并调整节奏。
5) **可视化进度**：至少提供页级与节点级的学习进度展示（播放游标）。

### 1.4 非功能性需求（指标/约束）
- 解析耗时：≤2 分钟/份（异步任务 + 限并发）。
- 问答体感：≤5 秒（SSE 流式 + 分句输出）。
- 准确率：知识点识别 ≥80%；答案准确率 ≥85%（溯源 + 本页优先 + 结构化召回）。
- 并发：≥10 人；服务器资源 8C16G。
- 集成：支持嵌入泛雅 Web/学习通 WebView；开放接口签名校验。

## 2. 总体架构设计

### 2.1 架构原则
- **独立演进 + 适配器模式（OpenAPI Adapter）**：
  - **原因**：内部 API 追求开发效率（RESTful + JWT）与 AI 深度（SSE 流式数据帧）；外部 API 追求兼容性。
  - **实现**：内部核心逻辑（Core Logic）是独立的，通过 `OpenAPI Adapter` 映射器层将泛雅规范（`enc` 签名、固定 `v1` 路径、特定响应结构）实时转换为内部处理流。
- **薄业务、重编排**：Go 侧负责鉴权、状态机、异步调度与流式代理；AI 复杂逻辑在 Python。
- **输出结构化**：AI 输出不仅是文本，还包括溯源、理解度、续接游标等“决策字段”。

### 2.2 组件与职责
- 前端（Vue）：教师端上传/编辑；学生端播放/打断/展示 SSE。
- Go API：
  - 课件存储（MinIO）与元数据（Postgres）
  - 解析任务队列（Redis）
  - 学习进度状态机（Redis）
  - SSE 问答代理（边到边转发）
  - 泛雅 OpenAPI 适配层（enc 签名、requestId、标准响应）
- Python AI 引擎：
  - 多模态解析（VLM）
  - 脚本生成（LLM）
  - RAG 问答（向量检索 + 溯源）
  - 意图识别与重讲判定（need_reteach/understandingLevel）

相关接口规范见：`API_DESIGN_V2.md`（内部+OpenAPI 适配层）。

## 3. 数据结构与关键模型

### 3.1 课件与解析任务
- `courseId`：课件 ID
- `parseId`：解析任务 ID（用于异步查询/生成脚本）
- `taskStatus`：processing/completed/failed

### 3.2 页级脚本（核心：续接能力的基础）
建议统一结构（可存 JSON 或拆表）：

```json
{
  "courseId": "...",
  "pageNum": 1,
  "pageSummary": "...",
  "nodes": [
    {"nodeId": "p1_n1", "type": "opening", "text": "...", "duration": 10},
    {"nodeId": "p1_n2", "type": "explain", "text": "...", "duration": 40},
    {"nodeId": "p1_n3", "type": "transition", "text": "...", "duration": 8}
  ]
}
```

### 3.3 学习会话与进度状态机（Redis 热数据）
- `sessionId`
- `currentPage/currentNodeId`
- `interrupted`
- `resumePage/resumeNodeId`
- `lastQaSummary`

## 4. 算法流程设计（重点体现 AI 融合深度）

> 说明：本节强调“AI 输出不仅是文本，而是可驱动教学流程的结构化结果”。

### 4.1 课件多模态解析（Parser）

**输入**：PPTX/PDF（文件 URL 或对象存储地址）

**流程**：
1. 拆页/渲染：PDF 按页渲染为图片；PPTX 按幻灯片导出图片（保留版式）。
2. 视觉结构化提取（VLM）：对每页输出：标题/要点/公式含义/图表结论/重点标注。
3. 标准化与校验：将模型输出映射到统一 schema（缺字段补全、长度/类型校验）。
4. 生成“结构预览”：chapters/subChapters/pageRange/isKeyPoint（用于平台侧快速预览）。

**输出**：页级结构化内容 + 知识点层级树 + 解析统计。

### 4.2 讲授脚本生成（Generator）

**目标**：生成符合教学逻辑的“可讲授脚本”，并可被前端高亮播放。

**流程**：
1. 以每页摘要/要点为输入，按模板生成：opening/explain/transition。
2. 输出节点化脚本：每段带 `nodeId/type/duration/keyPoints`。
3. （可选）生成导图：mindmap_markdown 或节点表。

**输出**：结构化脚本 `nodes[]` + 思维导图。

### 4.3 RAG 问答（Grounded QA）

**输入**：question + currentPage/currentNodeId +（可选）history

**流程**：
1. 意图识别（Intent）：判断是“正常提问”还是“没听懂/要重讲”。
2. 召回检索（Retrieval）：
   - 本页优先：检索 currentPage 的切片与摘要
   - 全局扩展：当本页召回不足再扩展全局
3. 生成（Generation）：将召回证据拼入 Prompt，要求输出带溯源字段。
4. 结果结构化：
   - `answer`
   - `sourcePage/sourceSection`
   - `understandingLevel`（none/partial/full）
   - `needReteach`
   - `resumePage/resumeNodeId`

**输出**：可解释、可溯源、可续接的问答结果。

### 4.4 进度续接与节奏调整（Policy）

**输入**：问答结果（understandingLevel/needReteach/resume*）+ 当前 session

**策略示例**：
- `full`：直接续讲 `resumeNodeId`。
- `partial`：返回 `adjustType=supplement`，补充 20~40 秒例子/重述，再续讲。
- `none`：返回 `adjustType=reteach`（或 supplement + 更慢语速），并建议回退至上一关键节点。

输出可直接对齐泛雅规范中的 `/api/v1/progress/adjust` 的 `adjustPlan`。

### 4.5 流式交互（SSE）
- Go 作为 SSE 代理，将 Python 的流式 token 边到边转发。
- 建议事件：`token/sentence/final/error`。
- 目标：首 token 快返回 + 分句触发 TTS，满足 ≤5 秒体感。

## 5. 泛雅平台集成方案（落地说明）

### 5.1 OpenAPI Adapter（对外）
严格对齐：
- 路径：`/api/v1/lesson/*`、`/api/v1/qa/*`、`/api/v1/progress/*`
- 签名：`enc=MD5(参数ASCII升序拼接 + staticKey + time)`
- 响应：`code/msg/data/requestId`

接口明细见：`API_DESIGN_V2.md` 第 8 章。

### 5.2 平台嵌入
- 学生端/教师端页面可作为泛雅内嵌 H5；通过平台侧用户身份信息建立 userId 映射。

### 5.3 异步与回调
- 解析/脚本生成：开放 API 返回 taskId（parseId/scriptId）+ taskStatus。
- 后续扩展：支持回调通知（新增 callbackUrl 参数，保持兼容）。

## 6. 评测与验收方案

### 6.1 数据集
- ≥100 份课件样本（文/理/工），覆盖 PDF/PPTX，包含公式与图表。

### 6.2 指标与验收
- 解析：
  - 成功率
  - 用时 ≤2min/份（限并发后测）
  - 知识点识别准确率 ≥80%（抽样人工标注对比）
- 问答：
  - 首 token ≤2s（体感）
  - 完整回答 ≤5s（网络良好）
  - 答案准确率 ≥85%（抽样人工判分 + 溯源一致性检查）

---

## 7. 风险与对策（工程化必备）

- 资源不足（8C16G）：解析限并发；大模型能力外部化；超时/重试。
- 幻觉与可信度：本页优先 + 溯源字段 + 结构化召回；必要时启用“拒答/提示翻页”。
- 对接复杂：双接口层；OpenAPI Adapter 严格对齐泛雅规范；文档同步为强约束。
