# Dify 融合架构重构版（前端交互 / 后端数据 / Dify AI）

## 0. 先给结论
这套系统完全可行，你的功能愿景也足够完整。
真正决定能否高质量落地的关键，不是功能数量，而是三层边界是否清晰：

1. 前端：负责交互体验与状态展示。
2. 后端：负责业务真相数据、权限、事务与一致性。
3. Dify：负责 AI 编排、推理、知识检索与生成。

这版文档的目标是把你的设计哲学落成可执行工程方案，降低推进失焦风险。

---

## 1. 架构原则（与你的设计哲学对齐）

### 1.1 五条硬规则
1. 单一事实源：业务主数据只在后端数据库落地。
2. AI 中台化：Dify 只承担 AI 工作流，不承担核心业务主存储。
3. 网关统一：前端不直连 Dify，统一通过后端调用。
4. 可观测可回放：每次 AI 调用有 request_id，可追踪、可复现。
5. 可降级：Dify 失败时，后端有兜底策略（缓存结果、模板输出、重试）。

### 1.2 分层职责表
| 层 | 必做职责 | 典型模块 | 不建议承担 |
|---|---|---|---|
| 前端 | 页面交互、流程引导、结果呈现 | 教师端/学生端/管理员端 | 权限判定真逻辑、数据主存储 |
| 后端 | 鉴权、业务规则、数据一致性、审计 | 用户/班级/课程/作业/成绩/笔记/错题 | 复杂 Prompt 编排 |
| Dify | 工作流编排、RAG、LLM 生成、结构化输出 | 解析、问答、题目生成、报告生成 | 业务主数据真相、强事务 |

---

## 2. 数据归属设计（避免后期崩盘）

## 2.1 后端数据库保存（主数据）
1. 用户与权限：user, role, tenant, class_membership。
2. 教学资产：course, chapter, knowledge_node, publish_record。
3. 学习过程：learning_progress, qa_record, note, wrong_question。
4. 作业考试：assignment, paper, submission, grading_result。
5. 运营统计：learning_event, class_metric_daily, user_profile_snapshot。

## 2.2 Dify 保存（AI 数据）
1. Workflow DSL 与版本。
2. 知识库索引与切片。
3. Prompt 模板、工具配置。
4. 执行日志、临时上下文。

## 2.3 数据回写原则
1. Dify 输出必须结构化 JSON。
2. 后端校验后再入库，不直接信任 LLM 原文。
3. 统一幂等键：request_id + tenant_id + scene。

---

## 3. 端到端主链路（可演示、可开发、可验收）

## 3.1 教师备课链路
1. 前端上传课件到后端。
2. 后端存文件与元数据，触发 Dify 解析工作流。
3. Dify 返回节点树与讲稿草稿。
4. 后端落库 version_1，前端进入可编辑态。
5. 教师确认发布，后端生成班级发布记录。

## 3.2 学生学习链路
1. 前端拉取课程节点树与学习进度。
2. 学生问答时，前端调用后端问答接口。
3. 后端组装上下文后调用 Dify 问答工作流。
4. Dify 返回答案与引用节点，后端写入 qa_record。
5. 前端展示答案、引用来源、追问入口。

## 3.3 作业与批改链路
1. 教师发作业，后端生成 assignment 与题集版本。
2. 学生提交后，客观题后端可先本地判分。
3. 主观题或解析由 Dify 评分工作流辅助。
4. 后端统一计算总分并落 grading_result。

## 3.4 学情反哺链路
1. 后端按日聚合学习事件。
2. 调用 Dify 学情分析工作流生成解释层文本。
3. 指标真值仍来自后端，Dify 只生成解读与建议。

---

## 4. Dify 工作流清单（按价值拆分）

## 4.1 P0 必须工作流（先跑通闭环）
1. WF-001 课件解析与节点树生成。
2. WF-002 讲稿生成与讲稿优化。
3. WF-003 知识点问答（课内检索优先）。
4. WF-004 习题生成（结构化 JSON 输出）。
5. WF-005 学情摘要与教学建议。

## 4.2 P1 增强工作流（拉开差距）
1. WF-006 错题归因分析。
2. WF-007 同类题推荐。
3. WF-008 个性化复习计划。
4. WF-009 学生学习报告生成。

## 4.3 P2 拓展工作流（时间充裕再做）
1. WF-010 多模型路由与 A/B 策略。
2. WF-011 内容审核与合规辅助。
3. WF-012 教师备课自动优化建议。

## 5. 接口契约（防止联调失控）

### 5.1 统一请求包络
```json
{
  "request_id": "uuid",
  "tenant_id": "t_001",
  "user_id": "u_001",
  "scene": "teacher.generate_questions",
  "input": {
    "course_id": "c_001",
    "node_ids": ["n_01", "n_02"]
  }
}
```

### 5.2 统一响应包络
```json
{
  "request_id": "uuid",
  "code": 0,
  "message": "ok",
  "data": {},
  "trace": {
    "dify_workflow_id": "wf_xxx",
    "latency_ms": 1320,
    "model": "qwen"
  }
}
```

### 5.3 强制校验
1. 后端对 data 做 JSON Schema 校验。
2. 校验失败写 error_log，不入主表。
3. 同 request_id 重放时必须幂等。

---

## 6. Dify 融合的关键架构设计

## 6.1 工作流分层模型（避免“大一统工作流”）
1. 编排层工作流：只做流程串联与路由，不做重计算。
2. 能力层工作流：只做单能力输出，如解析、问答、生成、评估。
3. 数据层回写：统一由后端落库，Dify 不直接写核心业务表。

这个分层会直接降低复杂度：当某一能力变更时，不会牵动整条业务链路。

## 6.2 工作流版本与兼容策略
1. 每个场景固定 scene 名称，scene 不随模型变化。
2. scene 绑定 workflow_id 与 version，避免“无版本漂移”。
3. 输出结构字段向后兼容，新增字段只追加不覆盖。

这样可保证前端与后端接口稳定，AI 能力升级不破坏业务层。

## 6.3 RAG 与知识边界
1. 课内问答：优先课程知识库检索，限制上下文来源。
2. 联网问答：必须在响应中标注外部来源，避免与教材内容混淆。
3. 教师生成内容：保留可编辑态，不直接覆盖原稿。

核心思想是“可控生成”，而不是“自由生成”。

## 6.4 网关化接入与可追踪性
1. 前端永远通过后端网关访问 AI。
2. 每次 AI 调用生成 request_id，并在前后端日志贯通。
3. 后端记录模型、延迟、token、重试次数等可观测数据。

这保证排障时能回答三个问题：谁触发、调用了什么、为什么失败。

---

## 7. 覆盖性评估（对应你的设计哲学）

## 7.1 已覆盖的核心诉求
1. 备课、学习、评估、反馈的闭环关系完整。
2. 教师、学生、管理员三角色职责边界清晰。
3. 学情回流到教师备课建议的逻辑成立。
4. AI 深度参与但不接管系统主数据。

## 7.2 当前模型下仍需显式定义的项
1. 多租户隔离：tenant_id 在接口、日志、存储全链路透传。
2. 指标口径：学习时长、掌握度、正确率计算公式要唯一。
3. 成本模型：按 scene 统计 token 与调用开销，支持审计。
4. 回滚机制：工作流版本不可用时自动回退上个稳定版本。

---

## 8. 架构风险与反模式

## 8.1 高风险反模式
1. 用 Dify 全局变量池承载班级/作业/成绩主数据。
2. 一个工作流同时处理权限、检索、生成、写库全流程。
3. 前端直接调用 Dify，绕过后端鉴权与审计。
4. 以自然语言自由输出代替结构化输出契约。

## 8.2 对应治理策略
1. 主数据只在后端数据库，Dify 仅保存 AI 上下文与索引。
2. 工作流单一职责，按场景拆分可复用原子能力。
3. 所有 AI 访问统一经后端网关，权限与配额集中控制。
4. 输出先过 JSON Schema 校验，校验失败不入主库。

---

## 9. 架构验收口径
1. 任一课程链路可追踪到完整 request_id 调用轨迹。
2. 任一 AI 结果都能定位对应 workflow_id 与 version。
3. Dify 不可用时，业务接口仍返回可解释兜底结果。
4. 学情指标的最终值可追溯到后端原始行为事件。
5. 角色越权访问在后端网关层可被拦截并审计。

---

## 10. 设计结论
这套方案保持了你最重要的设计哲学：功能全面、AI 深度融合、教学闭环完整。

同时通过三层边界把工程风险控制在可治理范围内：
1. 前端保证体验。
2. 后端保证真相。
3. Dify 保证智能。

最终目标不是减少功能，而是让每个功能都能稳定交付、可追踪、可演进。

---

## 11. 当前应用差距诊断（基于现状）
当前 Dify 应用已经跑通，但仍属于最小演示形态，核心问题不是“模型能力弱”，而是“链路设计过浅”。

### 11.1 当前链路问题
1. 当前主链路接近单路径：开始 -> LLM -> 直接回复。
2. 缺少场景分流，教师/学生/管理请求没有路由隔离。
3. 缺少知识检索门控，课内检索、泛化问答、兜底回复混在同一层。
4. 缺少结构化输出节点，导致后端难以稳定回写业务表。
5. 缺少评估与审计节点，无法形成“可解释、可复盘”的 AI 结果。

### 11.2 结论
当前应用适合演示“可问可答”，不满足“教学闭环系统”目标。

---

## 12. 重构目标（从演示链路升级为教学闭环链路）
重构后不是一条大工作流，而是“统一编排层 + 场景能力流”的体系。

### 12.1 统一编排层（入口网关流）
职责：
1. 解析 scene 与角色。
2. 执行权限与输入完整性校验。
3. 路由到对应能力流。
4. 统一输出 request_id、trace、结构化 data。

### 12.2 P0 五条能力流（必须具备）
1. 课件解析流：文件抽取 -> 节点化 -> 讲稿草案 -> 结构化输出。
2. 课堂问答流：问题改写 -> 课内检索 -> 回答生成 -> 引用补全 -> 可信度标注。
3. 题目生成流：知识点抽取 -> 题型规划 -> 题目生成 -> 答案与解析 -> JSON 校验。
4. 作业评估流：提交内容标准化 -> 评分规则匹配 -> 评语生成 -> 风险词审查。
5. 学情分析流：行为聚合输入 -> 弱点归因 -> 教学建议 -> 班级/个人双视角输出。

### 12.3 P1 三条增强流（形成差异化）
1. 错题归因与同类题推荐流。
2. 个性化复习计划流（按知识点掌握度）。
3. 教师备课优化建议流（基于历史学情）。

---

## 13. 单流内部标准结构（每条能力流统一）
每条流按六段式设计，降低维护复杂度：
1. 输入标准化：字段对齐、缺省补齐、敏感词预处理。
2. 上下文构建：课程、班级、节点、历史会话拼装。
3. 能力执行：LLM 或工具节点执行主推理。
4. 结构化包装：统一映射为业务 JSON。
5. 质量门控：规则校验、置信度评估、降级决策。
6. 追踪输出：trace、latency、model、source 引用回传。

---

## 14. 与后端契约的强约束
1. 每条流必须有固定 scene，不允许靠提示词文本隐式分流。
2. 每条流必须有 JSON Schema 对应，不允许自由文本直接入库。
3. 每条流必须输出 request_id 与 workflow_version。
4. 每条流必须定义失败降级分支（模板输出或缓存输出）。

---

## 15. 重构验收标准（可直接用于联调）
1. 至少 5 条 P0 工作流可独立运行并被统一编排层路由。
2. 任意一次调用都能在后端日志中用 request_id 追到完整链路。
3. 课堂问答必须返回引用来源，且可区分课内/课外来源。
4. 题目生成结果必须通过 Schema 校验后才能入库。
5. 学情分析必须输出可解释字段，不只输出主观建议文本。

---

## 16. 下一步落地顺序（执行优先级）
1. 先完成统一编排层与 scene 路由。
2. 再上线问答流与题目生成流（高频核心路径）。
3. 随后补齐课件解析流与学情分析流。
4. 最后接入错题推荐与复习计划等增强流。

该顺序可以保证每一阶段都可演示、可验收、可持续扩展。
