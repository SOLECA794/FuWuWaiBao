### 二、重构后的 Dify 全链路编排方案（完全可落地，解决你没想清的核心难点）

核心设计思路：**双端解耦 + 3 条核心主工作流 + 2 个公共能力池 + 全局状态管理**，所有逻辑完全基于 Dify 原生功能实现，无需额外开发，每个工作流对应前端一个独立 API，时序清晰，可单独调试、单独导出 DSL。

#### 整体架构总览

plaintext

```
┌── 教师端工作流集群（3条，异步执行，仅教师可触发）
│   ├── 1. 课件解析与节点树初始化工作流（上传课件时触发）
│   ├── 2. 节点与讲稿编辑同步工作流（教师修改内容时实时触发）
│   └── 3. 课件发布与批量预生成工作流（教师点击发布时触发，核心主线）
├── 学生端工作流集群（2条，同步执行，仅学生可触发）
│   ├── 1. 课程学习与断点续接控制工作流（核心，解决打断续接难点）
│   └── 2. 学习进度关联式问答工作流（匹配学生提问需求）
├── 公共能力池（可复用原子节点，所有工作流通用）
│   ├── 知识点语义检索模块
│   ├── LLM生成模块
│   ├── TTS语音生成模块
│   ├── 可控联网搜索模块
└── Dify全局变量池（状态管理核心，全工作流数据同步）
    ├── 课件节点树全局存储（按course_id绑定）
    ├── 学生学习进度状态（按user_id+course_id绑定）
    ├── 讲课断点状态（按user_id+course_id绑定）
```

### 三、每条工作流的详细编排设计（Dify 里直接拖拽可实现）

#### ▶️ 教师端核心工作流（3 条，完全匹配教师操作路径）

##### 1. 课件解析与节点树初始化工作流

（教师上传课件时触发，生成初始节点树，给教师提供编辑基础）

表格


| 节点顺序                 | 节点配置详情                                                                                                                                                                                                                                           |
| :----------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 开始节点                 | 输入变量：`{{course_file}}`教师上传的课件文件`{{course_id}}`前端生成的课件唯一 ID`{{teacher_id}}`教师唯一 ID                                                                                                                                           |
| 文档解析节点             | 输入：`{{course_file}}`配置：开启「提取页码、章节标题、文本格式」，关闭「自动摘要」输出：带页码标记的课件全文本                                                                                                                                        |
| 语义分段与节点树生成节点 | 输入：解析后的全文本核心配置：1.  语义分段：最大长度 1024 字符，重叠 100 字符，绑定对应页码2.  调用 LLM（glm-4-plus），自动生成**不固定层级的结构化节点树 JSON**，教师后续可自由编辑3.  每个知识点自动生成初始讲稿，绑定唯一 point\_id、页码、原文内容 |
| 全局变量存储节点         | 把生成的节点树 JSON 存入 Dify 全局变量池，和`{{course_id}}`永久绑定                                                                                                                                                                                    |
| 课件临时知识库创建节点   | 为该课件创建临时编辑用知识库，每个知识点作为独立文档存入，绑定 point\_id 元数据                                                                                                                                                                        |
| 结束节点                 | 输出：完整节点树 JSON、临时知识库 ID前端拿到后直接渲染编辑界面，教师可开始修改                                                                                                                                                                         |

##### 2. 节点与讲稿编辑同步工作流

（教师在前端修改内容后实时触发，无需重跑全流程，仅更新修改的部分）

表格


| 节点顺序           | 节点配置详情                                                                                                                                            |
| :----------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 开始节点           | 输入变量：`{{course_id}}`课件 ID`{{update_type}}`修改类型（节点结构 / 讲稿内容）`{{target_point_id}}`被修改的知识点 ID`{{updated_content}}`修改后的内容 |
| 节点树更新节点     | 从全局变量池读取对应节点树，按修改类型更新对应内容，重新存入全局变量池                                                                                  |
| 临时知识库同步节点 | 同步更新临时知识库中对应知识点的内容，保证后续生成用教师修改后的最新内容                                                                                |
| 结束节点           | 输出更新后的节点树，前端实时刷新编辑界面                                                                                                                |

##### 3. 课件发布与批量预生成工作流

（教师点击「发布」时触发，核心主线，一次性完成全量内容生成，发布后学生端可见）

表格


| 节点顺序             | 节点配置详情                                                                                                              |
| :------------------- | :------------------------------------------------------------------------------------------------------------------------ |
| 开始节点             | 输入变量：`{{course_id}}`待发布的课件 ID                                                                                  |
| 发布前校验节点       | 校验节点树是否完整、讲稿是否无空内容，校验不通过直接终止并返回报错，避免发布残缺内容                                      |
| 正式知识库固化节点   | 把临时编辑用知识库，复制为**正式发布知识库**，永久固化内容，后续教师编辑仅修改临时库，不影响已发布的学生端内容            |
| 全量讲稿批量优化节点 | 遍历节点树的所有知识点，调用 LLM 批量优化教师编辑后的讲稿，适配语音播报（断句、语气、口语化优化），保证音频生成效果       |
| 全量音频批量生成节点 | 遍历优化后的所有讲稿，批量调用 TTS 生成对应音频，每个音频绑定`point_id`，生成「音频链接 + 讲稿文本 + 对应时间戳」的映射表 |
| 全局状态更新节点     | 1.  把固化后的节点树、音频映射表存入全局变量池，标记为「已发布」2.  把课件 ID 加入学生端可见列表                          |
| 结束节点             | 输出：发布成功状态、正式知识库 ID、学生端访问参数前端提示教师发布成功，学生端可正常访问该课程                             |

#### ▶️ 学生端核心工作流（2 条，解决你最关心的「打断续接」+「关联式问答」核心需求）

##### 1. 课程学习与断点续接控制工作流（核心难点，100% 可落地实现）

**解决的核心问题**：预生成的讲课音频被学生提问打断后，如何智能续接，实现「数字人讲课→实时打断→问答→续接讲课」的丝滑体验。

表格


| 节点顺序                 | 节点配置详情                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| :----------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 开始节点                 | 输入变量（前端实时传入）：`{{course_id}}`当前学习的课件 ID`{{user_id}}`学生唯一 ID`{{action_type}}`动作类型（开始播放 / 暂停 / 打断 / 续接）`{{current_point_id}}`当前播放的知识点 ID`{{current_timestamp}}`当前播放的音频时间戳（毫秒）                                                                                                                                                                                                                                                                                                                                                                                  |
| 全局状态同步节点         | 把学生的「当前学习进度（current\_point\_id）」「播放断点（timestamp）」实时存入 Dify 全局变量池，按`user_id+course_id`绑定，为后续问答、续接提供数据基础                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| 分支判断节点（核心逻辑） | 按`action_type`分 4 个分支处理：✅ 分支 1：开始播放→ 从全局变量池读取该课件的节点树、音频映射表，返回给前端，前端从第一个知识点开始播放✅ 分支 2：暂停→ 仅更新全局断点状态，无额外操作，前端暂停播放✅ 分支 3：打断（学生发起提问时触发）→ 锁定当前断点状态，触发「关联式问答工作流」，前端暂停播放，等待问答结果✅ 分支 4：续接（问答结束后触发）→ 核心续接逻辑：1.  从全局变量池读取之前锁定的断点（知识点 ID + 时间戳）2.  调用 LLM 生成一句丝滑的过渡语，比如「刚才我们讲到了 XX 知识点，接下来我们继续学习」3.  把过渡语实时生成短音频，和断点后的原音频拼接，返回给前端4.  前端从断点位置继续播放，实现无感续接 |
| 结束节点                 | 输出对应动作的结果（音频列表、断点状态、过渡音频、问答触发信号），前端执行对应播放操作                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |

##### 2. 学习进度关联式问答工作流

（学生提问时触发，匹配「学习进度关联 + 学生自主控制联网」的需求）

表格


| 节点顺序               | 节点配置详情                                                                                                                                                                                                                                                                                                 |
| :--------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 开始节点               | 输入变量：`{{course_id}}`课件 ID`{{user_id}}`学生 ID`{{user_query}}`学生的提问内容`{{allow_online_search}}`学生选择的是否允许联网（true/false）                                                                                                                                                              |
| 学习进度上下文获取节点 | 从全局变量池读取该学生的当前学习进度`current_point_id`，自动获取「该知识点 + 前后相邻 3 个知识点」的内容，作为「学习进度上下文」，保证回答贴合学生的学习节奏                                                                                                                                                 |
| 全课件知识点检索节点   | 从正式发布知识库中，检索和用户提问匹配的 Top3 相关知识点，作为「全课件关联上下文」                                                                                                                                                                                                                           |
| 上下文合并节点         | 把「学习进度上下文」+「全课件关联上下文」合并，作为最终喂给 LLM 的参考内容，实现你要的「附近知识点 + 整体知识点关联回答」                                                                                                                                                                                    |
| 联网控制分支节点       | 按学生选择的`allow_online_search`分分支：✅ 分支 1：false（仅课件内容）→ 仅用合并后的课件上下文喂给 LLM，强制关闭联网，系统提示词要求 100% 基于课件内容回答✅ 分支 2：true（允许联网）→ 先判断课件上下文的匹配度，匹配度≥0.6 优先用课件内容，不足的部分用联网搜索内容补充，强制标注「拓展内容来自互联网」 |
| LLM 生成节点           | 模型：glm-4.5-air，参数：温度 0.1、Top-P 0.3，保证输出稳定贴合上下文                                                                                                                                                                                                                                         |
| TTS 语音生成节点       | 把回答生成对应音频，适配学生端的语音播报需求                                                                                                                                                                                                                                                                 |
| 结束节点               | 输出：文本回答、对应知识点 ID、课件页码、回答音频，前端渲染展示，同时给续接工作流发送「问答完成」信号                                                                                                                                                                                                        |

### 四、你没考虑到的细节补充 & 兜底方案

1. **教师修改已发布课件的处理逻辑**

   教师发布后如果需要修改，仅修改临时编辑库的内容，修改完成后可选择「更新发布」，系统会增量生成修改部分的讲稿和音频，无需全量重新生成，不影响学生端的历史学习进度。
2. **学生学习进度持久化**

   学生每次退出课程时，自动把当前学习进度存入全局变量池，下次进入课程时，可直接从上次退出的知识点继续播放，无需从头开始。
3. **异常兜底方案**

   * 音频播放失败：自动返回对应讲稿的文本内容，保证学生学习不中断
   * 问答接口超时：自动重试 3 次，仍失败返回固定兜底话术，不影响讲课流程
   * 续接失败：自动返回断点对应的知识点完整音频，前端从头播放该知识点，避免流程卡死
